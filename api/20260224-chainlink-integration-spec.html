2. Why These Products, In This Configuration

2.1 CRE as the Settlement Orchestrator

The problem: An information market must resolve -- someone or something must determine whether "event X occurred by date Y" is true or false. This resolution must be verifiable by all participants and resistant to manipulation by any single party.

Why CRE (not a custom oracle): CRE provides decentralized consensus on workflow execution. When our market resolution workflow runs, it executes across multiple independent Chainlink nodes. The result is not "BlindOracle says YES" -- it is "the Chainlink node network reached consensus that the answer is YES." This is a fundamentally different trust model than a self-hosted oracle.

How we use it: Our UnifiedPredictionSubscription.sol contract implements the IReceiver interface [1]. When a market creator calls requestSettlement(), the contract emits a SettlementRequested event. A CRE Log Trigger workflow detects this event, reads the market data via EVM Read, queries AI providers via HTTP (Chainlink Functions), and calls onReport() with the verified outcome. The contract then enters a 24-hour dispute window before finalization.

What we add that CRE does not provide: The privacy layer. CRE resolves what happened. It does not know who bet on what. Positions are committed as SHA256 hashes -- SHA256(secret || position || amount) -- before market resolution. CRE verifies the outcome; the commitment scheme hides the participants.

2.2 CCIP as the Cross-Chain Settlement Rail

The problem: An agent on Base wants to settle a position funded from Arbitrum. Without a cross-chain protocol, this requires a centralized bridge -- a single point of failure and a trust assumption.

Why CCIP (not a third-party bridge): CCIP has an independent Risk Management Network that monitors cross-chain transactions [2]. This is an additional validation layer beyond the CCIP nodes themselves. For a financial settlement system handling agent funds, the extra validation layer justifies the choice over cheaper but less-validated alternatives.

How we use it: Our ccip_btc.py module and chainlink_integration.py configure CCIP routers for Ethereum, Arbitrum, Base, Polygon, Optimism, and Avalanche. When an agent requests a cross-rail transfer (via the /v2/transfer/cross-rail API endpoint), the settlement engine routes through CCIP for any chain-to-chain movement.

What we add: A privacy hop before the cross-chain transfer. Funds enter a guardian federation and emerge as blind-signed tokens. Only then does the cross-chain transfer happen. CCIP sees a transfer from a federation address, not from an identified agent. The privacy layer wraps the interoperability layer.

2.3 Data Streams + Functions for Multi-AI Market Resolution

The problem: Information markets ask questions like "Will event X occur by date Y?" Resolving these requires: (a) objective data (price feeds, event data) and (b) subjective interpretation (did the event meet the resolution criteria?).

Why both Data Streams and Functions: Data Streams handle (a) -- objective, quantitative data delivered at sub-second latency [3]. Functions handle (b) -- we send JavaScript source code to the DON that queries multiple AI providers (Gemini, GPT-4, Claude) to interpret whether the resolution criteria are met [4]. The AI models analyze the question, the evidence, and the criteria independently. Our CRE workflow aggregates their responses and requires consensus (2-of-3 minimum) before accepting an outcome.

What we add: The multi-AI consensus layer. Chainlink Functions executes the code; we designed the consensus protocol that requires independent agreement across AI providers. This is Byzantine fault-tolerant AI verification -- a single hallucinating or compromised model cannot determine a market outcome.

2.4 Confidential Compute Alignment (Future)

The opportunity: Chainlink Confidential Compute, announced at SmartCon 2025 and currently in early access [5], enables private computation in TEEs where data is hidden from node operators. This directly complements our blind signature privacy model.

Current state: Our PrivateClaimVerifier.sol contract is designed to receive privacy-preserving claim verifications. Today, it works with our application-layer blind signatures. When Confidential Compute reaches GA, the oracle query itself can be private -- the question "did event X occur?" can be answered without any node seeing the query content. This creates end-to-end privacy: private query (Confidential Compute) + private position (blind signatures) + private settlement (guardian federation).

Alignment with DECO: Chainlink's DECO protocol [6] enables proving statements about TLS data in zero-knowledge. For markets that resolve based on web-sourced data (news events, API responses), DECO can prove the data came from a specific source without revealing what the data says to the oracle nodes. This is a natural extension of our privacy architecture.

4.1 Settlement Lifecycle (On-Chain)

OPEN
  --> Creator calls requestSettlement()
  --> SettlementRequested event emitted

SETTLEMENT_REQUESTED
  --> CRE Log Trigger activates
  --> Workflow: EVM Read (market data) + HTTP (AI queries) + Consensus
  --> onReport() called with verified outcome

PENDING_FINALIZATION
  --> 24-hour dispute window begins
  --> Dispute requires 10% pool stake + evidence
  --> If disputed: CRE re-verification triggered
  --> If no dispute after 24h: auto-finalize

SETTLED
  --> Winners claim proportional ETH from pool
  --> Privacy layer: claims use blind-signed tokens (unlinkable to position)
  --> Cross-chain payouts via CCIP if needed

5.3 Why Solid (Security Model)

Every component in the stack uses a proven primitive with known security properties:

  • SHA256 commitment scheme: NIST FIPS 180-4 [11]. Computational hiding (preimage resistance ~2256), computational binding (collision resistance ~2128). Standard assumption, no novel cryptography.
  • Chaumian blind signatures: Chaum 1982 [12]. Information-theoretic unlinkability between deposit and token. 44 years of peer review.
  • CRE decentralized consensus: Multi-node execution with OCR aggregation. Production since November 2025 with institutional deployments (UBS, Swift, Mastercard) [7].
  • CCIP Risk Management Network: Independent validation layer monitoring cross-chain transactions [2]. Defense-in-depth for the bridge.
  • Byzantine consensus (CaMel Layer 2): 67% threshold for standard operations, 80% for high-value. Based on Lamport, Shostak, and Pease (1982) [13] and Castro and Liskov's pBFT (1999) [14].
  • MASSAT 87-test security assessment: Automated, scheduled, with on-chain proof NFT. Results independently verifiable. See MASSAT results.

6. Payment Architecture: x402 + CRE Subscriptions

BlindOracle uses two payment models simultaneously:

6.1 x402 Micropayments (API Access)

Every API call is gated by HTTP 402 [8]. No API keys, no registration for anonymous-tier access. The agent pays per request via the X-402-Payment header. This is implemented in distribution/x402_api_gateway.py.

6.2 On-Chain Subscriptions (CRE Workflow Access)

Higher-tier access (premium data, higher position limits, enterprise features) uses OracleSubscription.sol with ERC20 payments (USDC, USDT, DAI, LINK). Subscriptions auto-renew via CRE CRON workflows. This is implemented in services/cre/cre_subscription_client.py.

6.3 Privacy-First Default

The default payment rail is blind-signed tokens via the guardian federation. The X-Payment-Rail header defaults to private. Transparent rails (stablecoin, fiat) are opt-in. See distribution/x402_api_gateway.py.
