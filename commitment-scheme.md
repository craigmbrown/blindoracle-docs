## Abstract

This document specifies the SHA256 commitment scheme used by BlindOracle for privacy-preserving position placement in information markets. The scheme provides both hiding (positions are secret until reveal) and binding (positions cannot be changed after commitment).

## 1. Commitment Construction

A commitment `C` is constructed as:

```
C = SHA256(secret || position || amount)
```

Where:

| Field | Type | Size | Description |
|---|---|---|---|
| `secret` | bytes | 32 bytes | Cryptographically random value generated by the agent |
| `position` | string | Variable | Market outcome identifier (e.g., "YES", "NO", or a numeric value) |
| `amount` | string | Variable | Settlement units as a decimal string (e.g., "0.10") |
| `\|\|` | - | - | Concatenation operator |

The concatenation is performed on the raw byte representations. `position` and `amount` are UTF-8 encoded before concatenation.

## 2. Security Properties

### 2.1 Hiding

Given only `C`, an observer cannot determine `position` or `amount`.

**Proof sketch**: SHA256 is a preimage-resistant function per NIST FIPS 180-4 [1]. The 256-bit `secret` ensures the input space is `2^256` times larger than the (small) position space. Even when `position` is drawn from a binary set {"YES", "NO"}, an attacker must find the 32-byte secret to recover the position. This requires `O(2^256)` operations, which is computationally infeasible.

### 2.2 Binding

Once `C` is published on-chain, the agent cannot later claim a different `(secret, position, amount)` triple.

**Proof sketch**: Finding two distinct inputs that produce the same SHA256 output requires breaking collision resistance. The best known attack on SHA256 requires `O(2^128)` operations [2]. No practical collision has been demonstrated.

### 2.3 Non-Linkability (with blind signatures)

When combined with Chaumian blind-signed tokens [3], the commitment provides deposit-position unlinkability:

1. Agent deposits value to guardian federation, receives blind-signed tokens
2. Agent uses tokens to fund commitment `C` -- the federation cannot link the tokens to the original deposit
3. Commitment `C` is published on-chain -- the smart contract cannot link `C` to the depositor
4. The information-theoretic gap between deposit identity and position identity is maintained

## 3. Reveal Protocol

At market resolution time, the agent reveals `(secret, position, amount)`:

```
verify(C, secret, position, amount):
    computed = SHA256(secret || position || amount)
    return computed == C
```

The smart contract performs this verification on-chain. If the commitment matches and the position is correct per the oracle resolution, the agent receives the payout.

## 4. Implementation Notes

### 4.1 Secret Generation

The `secret` MUST be generated using a cryptographically secure random number generator (CSPRNG). Using a predictable or low-entropy secret defeats the hiding property.

```python
import secrets
secret = secrets.token_bytes(32)
```

### 4.2 Canonical Encoding

To ensure deterministic commitment values:

- `position` is lowercase, trimmed of whitespace
- `amount` is a decimal string with no trailing zeros (e.g., "0.1" not "0.10")
- Concatenation order is fixed: secret first, then position, then amount
- All strings are UTF-8 encoded

### 4.3 On-Chain Verification

The Solidity verification uses `keccak256` (available natively) or `sha256` (precompile at address 0x02):

```solidity
function verifyCommitment(
    bytes32 commitment,
    bytes32 secret,
    string calldata position,
    string calldata amount
) public pure returns (bool) {
    bytes32 computed = sha256(abi.encodePacked(secret, position, amount));
    return computed == commitment;
}
```

## 5. Threat Model

| Threat | Mitigation | Residual Risk |
|---|---|---|
| Secret brute-force | 256-bit entropy | Negligible (2^256 operations) |
| Position guessing (binary market) | Secret hides position | Cannot verify guess without secret |
| Commitment malleability | SHA256 is collision-resistant | No known collision (2^128 bound) |
| Front-running | Commitment hides position before reveal | Position unknown until market resolves |
| Replay attack | Commitments are bound to market ID | Cannot reuse commitment across markets |
| Oracle manipulation | CRE decentralized consensus + multi-AI verification | Byzantine fault tolerance (67% threshold) |

## 6. Comparison with Alternative Schemes

| Scheme | Hiding | Binding | On-Chain Cost | Complexity |
|---|---|---|---|---|
| **SHA256 commitment** (this spec) | Yes | Yes | Low (~50K gas) | Simple |
| Pedersen commitment | Yes (information-theoretic) | Yes (computational) | Medium (~100K gas) | Moderate |
| ZK-SNARK proof | Yes | Yes | High (~300K gas) | High |
| Simple hash (no secret) | No | Yes | Low | Simple (insecure) |

BlindOracle uses SHA256 commitments because they provide adequate security with minimal on-chain cost and implementation complexity. The hiding property is computational (not information-theoretic), but the 256-bit secret makes the computational gap irrelevant in practice.
